# Tax Portal Integration Architecture

This document outlines the technical design, security workarounds, and systemic flow of the `integration.taxportal` module within the **Dynorix Gaime Bridge**.

## 1. Overview

The primary challenge of extracting electronic invoices from the Azerbaijan e-Taxes portal (`new.e-taxes.gov.az`) is bypassing the portal's Web Application Firewall (WAF) and modern Single Page Application (SPA) anti-bot protections. 

We abandoned raw HTTP `fetch()` scraping in favor of an **SPA Interception Strategy** powered by Playwright. This ensures that the portal's native frontend code manages the complex cryptographic headers, Axios interceptors, and JWT lifecycles, while our backend simply "listens" to the verified network traffic.

## 2. Session Management & ASAN Imza

The e-Taxes portal enforces a strict two-token swap protocol during the ASAN Imza mobile login.

### The Two-Token Swap
1. **Initial Verification**: The operator submits their Phone Number and User ID. Playwright clicks the verification button.
2. **Temporary JWT**: Upon successful mobile push confirmation, the portal grants a *temporary JWT* to load the dashboard.
3. **Company JWT**: Once the dashboard resolves the user's legal entity, the SPA seamlessly overwrites the temporary JWT in `localStorage` with a permanent `bridge-company-jwt` (or equivalent token).

**Our Persistence Strategy:**
Our code (`waitForLandingPageReady`) loops and waits up to 30 seconds for the permanent JWT to appear in `localStorage`. Once validated via an API ping (`/v2/profile`), we instruct Playwright to persist the entire browser context (cookies & `localStorage`) into `.run/playwright-tax-portal-state.json`. 

On subsequent syncs, we hydrate the browser with this state file. If the token is valid, we completely bypass the ASAN Imza mobile push requirement.

## 3. The SPA Interception Strategy

Fetching the paginated lists of `Incoming` and `Outgoing` invoices directly via injected `fetch()` calls fails with `400 Bad Request` because the WAF detects missing SPA-specific Axios interceptor headers.

### How We Intercept the List Data:
1. **Navigation**: We strictly navigate the Playwright browser to the physical UI routes (`/eportal/invoice`).
2. **Tab Simulation**: We simulate a UI click on the "Göndərilənlər" (Outgoing) or "Daxil olanlar" (Incoming) tabs.
3. **Eavesdropping**: Using `page.waitForResponse()`, we capture the backend's JSON response to the SPA's native XHR requests. 

This approach completely circumvents WAF blocks because the traffic is generated by the official frontend.

### Pagination Handover
The first page is captured natively. For pages 2 through N, we inject an `XMLHttpRequest` directly into the `window` scope (`executeJsonRequest`). Because it is executed within the active DOM window, the SPA's global Axios interceptors automatically attach the secure headers for us, preventing `400` errors on paginated loops.

## 4. Metadata Extraction and `403 Forbidden` Tolerance

To construct the canonical `TaxDocument` entity, we must query the detail level of each invoice. 

The e-Taxes v2 API aggressively restricts metadata:
- **`GET /.../invoice/{id}`**: Returns the core invoice details (Success `200 OK`).
- **`GET /.../invoice/{id}/versions`**: Often returns `403 Access Denied`.
- **`GET /.../invoice/{id}/history`**: Often returns `403 Access Denied`.
- **`GET /.../invoice/{id}/tree`**: Often returns `403 Access Denied`.

### Defensive Catching
Instead of aborting the multi-document synchronization phase when a metadata sub-request fails, the `PlaywrightTaxPortalClient` wraps each supplementary fetch in a `try-catch` block. It logs a soft warning for the missing data and allows the `TaxPortalSyncedDocument` to be constructed using just the primary details, ensuring resilient, interruption-free operation over hundreds of documents.

## 5. Development & Live Execution

### Official Run Profile (`dev`)
The easiest way to run the application for localized development is via the helper script:
```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\dev-first-run.ps1
```
This mounts local PostgreSQL databases, provisions demo data, and keeps the server hot for REST API interaction.

### Live Extraction Verification
To verify the token mechanics against the real portal:
1. Populate `.env` with `TAX_PORTAL_USERNAME` (phone), `TAX_PORTAL_PASSWORD` (id), and `TAX_PORTAL_LEGAL_TIN`.
2. Run the live verification:
```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-live-asan.ps1
```

## 6. Known Limitations
- The integration heavily relies on the current CSS selectors of the `eportal/invoice` page for tab switching. If the UI framework changes, the `documentInboxTabSelector` and `documentOutboxTabSelector` properties will need updates.
- Playwright downloads Chromium binaries on the first run; ensure transparent proxy access if running inside a corporate VPN.
